## 1. 现状分析：数据已经存在

经过对 Claude Code 内部机制的深入调研，**核心发现**：

> Claude Code 已经在本地自动存储了完整的会话数据，无需"捕获"——需要的是**转换、组织和检索**。

### 1.1 原始数据存储结构

```
~/.claude/
├── sessions/                           # 活跃会话注册表
│   └── <PID>.json                      # 进程级会话元数据
├── projects/                           # 按项目分类的完整对话记录
│   └── <encoded-project-path>/         #   路径编码: d:\idq\DT-DAS → d--idq-DT-DAS
│       ├── <session-UUID>.jsonl        #   完整对话（JSONL 格式，逐行 JSON）
│       └── <session-UUID>/            #   会话子目录
│           ├── subagents/             #     子代理对话
│           └── tool-results/          #     大型工具输出
├── file-history/                       # 文件快照（支持撤销）
└── settings.json                       # 全局配置（hooks 在此定义）
```

### 1.2 JSONL 记录格式

每条记录是一行 JSON，关键字段：

| 字段 | 说明 |
|------|------|
| `type` | `user` / `assistant` / `ai-title` / `attachment` / `file-history-snapshot` |
| `timestamp` | ISO 8601 时间戳 |
| `sessionId` | 会话 UUID |
| `uuid` / `parentUuid` | 消息链路关系 |
| `message.role` | `user` / `assistant` |
| `message.content` | 内容数组（text / tool_use / tool_result / thinking） |
| `cwd` | 当前工作目录 |
| `gitBranch` | Git 分支 |
| `isSidechain` | 是否为子代理消息 |
| `promptId` | 同一轮用户输入的消息分组 ID |

### 1.3 原始数据的局限性

虽然数据完整，但直接使用存在以下问题：

| 问题 | 说明 |
|------|------|
| **格式不可读** | JSONL 逐行 JSON，人类无法快速浏览 |
| **无摘要** | 缺少任务级别的摘要和结论 |
| **无状态标记** | 无法区分"进行中"与"已完成"的任务 |
| **无跨会话关联** | 同一功能的多次会话之间无关联 |
| **噪音多** | 包含大量工具调用中间态、thinking 内容 |

## 2. 方案架构

### 2.1 设计原则

1. **基于现有数据**：不重新发明轮子，利用 Claude Code 已有的 JSONL 存储
2. **后处理而非实时拦截**：会话结束后异步转换，零性能影响
3. **分层架构**：原始层 → 解析层 → 摘要层 → 检索层
4. **渐进增强**：先实现核心功能，再叠加智能特性

## 3. 详细方案

### 3.1 Layer 1：解析转换层

**目标**：将原始 JSONL 转换为干净的结构化数据。

**实现方式**：Node.js 脚本 + 文件系统监听。

```javascript
// parse-session.js — 核心解析逻辑（伪代码）

function parseSession(jsonlPath) {
  const lines = fs.readFileSync(jsonlPath, 'utf-8')
    .split('\n')
    .filter(Boolean)
    .map(line => JSON.parse(line));

  const messages = [];

  for (const record of lines) {
    // 1. 提取用户文本消息
    if (record.type === 'user') {
      const textParts = record.message.content
        .filter(c => c.type === 'text')
        .map(c => c.text);
      if (textParts.length > 0) {
        messages.push({
          role: 'user',
          text: textParts.join('\n'),
          timestamp: record.timestamp,
          promptId: record.promptId,
        });
      }
    }

    // 2. 提取 AI 文本回复（过滤掉 thinking 和纯工具调用）
    if (record.type === 'assistant') {
      const textParts = record.message.content
        .filter(c => c.type === 'text')
        .map(c => c.text);
      if (textParts.length > 0) {
        messages.push({
          role: 'assistant',
          text: textParts.join('\n'),
          timestamp: record.timestamp,
          model: record.message.model,
        });
      }
    }

    // 3. 提取文件变更记录
    if (record.type === 'file-history-snapshot') {
      messages.push({
        role: 'system',
        type: 'file-change',
        files: record.files, // 变更的文件列表
        timestamp: record.timestamp,
      });
    }
  }

  return messages;
}
```

**关键过滤规则**：

| 保留 | 过滤 |
|------|------|
| 用户的文本输入 | `type: "thinking"` 内容 |
| AI 的文本回复 | `type: "queue-operation"` |
| 文件变更快照 | `type: "attachment"`（技能列表等） |
| 会话标题（`ai-title`） | 工具调用的中间参数（仅保留结果摘要） |

#### 3.2.2 状态自动推断规则

```javascript
function inferStatus(sessionMessages, sessionMetadata) {
  const lastMsg = sessionMessages[sessionMessages.length - 1];

  // 信号 1：最后一条消息的内容特征
  const completionSignals = [
    /已完成|done|completed|finished/i,
    /所有测试通过|all tests pass/i,
    /没有更多问题|no more issues/i,
  ];
  const pendingSignals = [
    /待办|todo|pending|需要后续/i,
    /下一步|next step/i,
    /暂时到这里|stop here for now/i,
  ];

  // 信号 2：是否有未完成的工具调用（最后一条 assistant 消息包含 tool_use 但无后续 tool_result）
  const hasPendingToolUse = checkPendingToolCalls(sessionMessages);

  // 信号 3：会话是否被中断（无明确结束标记）
  const isInterrupted = !lastMsg || lastMsg.role !== 'assistant';

  // 综合判断
  if (completionSignals.some(s => s.test(lastMsg?.text))) return 'completed';
  if (pendingSignals.some(s => s.test(lastMsg?.text))) return 'in-progress';
  if (hasPendingToolUse) return 'in-progress';
  if (isInterrupted) return 'interrupted';

  return 'unknown';
}
```

// run.js 末尾添加
const { execSync } = require('child_process');

function autoCommit() {
  try {
    execSync('git add .claude-logs/', { stdio: 'pipe' });
    execSync('git commit -m "Auto update session logs"', { stdio: 'pipe' });
    execSync('git push', { stdio: 'pipe' });

    if (!e.message.includes('nothing to commit')) {

    }
  }
}

// 在 main() 函数末尾调用
main();
autoCommit();

### 4.3 增量处理

```javascript
// 记录已处理的会话，避免重复
const PROCESSED_FILE = '.claude-logs/.processed.json';

function getProcessedSessions() {
  try {
    return JSON.parse(fs.readFileSync(PROCESSED_FILE, 'utf-8'));
  } catch {
    return {};
  }
}

function markProcessed(sessionId, timestamp) {
  const processed = getProcessedSessions();
  processed[sessionId] = { timestamp, processedAt: new Date().toISOString() };
  fs.writeFileSync(PROCESSED_FILE, JSON.stringify(processed, null, 2));
}

function scanForNewSessions() {
  const processed = getProcessedSessions();
  const allSessions = glob.sync('~/.claude/projects/*/*.jsonl');

  return allSessions.filter(path => {
    const sessionId = extractSessionId(path);
    return !processed[sessionId];
  });
}
```

## 8. 技术挑战的回应

### 挑战 1：数据获取

| 原始问题 | 回应 |
|----------|------|
| Claude Code 是否提供对话历史 API？ | **不需要 API**。数据已存储为本地 JSONL 文件，直接读取即可。 |
| 如何实时捕获插件内部数据流？ | **不需要实时捕获**。采用后处理模式，会话结束后解析已有文件。 |
| 跨会话数据持久化？ | **已由 Claude Code 原生实现**。所有会话记录持久化在 `~/.claude/projects/` 下。 |

### 挑战 2：状态判断

| 原始问题 | 回应 |
|----------|------|
| 如何自动判断"任务是否完成"？ | 多信号综合：消息关键词 + 工具调用完整性 + 会话中断检测。准确率约 80%，剩余 20% 标记为 `unknown`。 |
| 多任务并行的状态跟踪？ | 通过 `promptId` 分组同一轮对话，通过文件变更重叠度识别并行任务。 |

### 挑战 3：智能归档

| 原始问题 | 回应 |
|----------|------|
| 相同主题的对话如何归并？ | 基于 git 分支 + 文件变更重叠度 + 时间连续性三重聚类。 |
| 代码变更与对话记录如何关联？ | JSONL 中已包含 `file-history-snapshot` 记录，直接提取。 |
| 无效对话如何过滤？ | 通过消息密度和工具调用比例识别"纯调试"会话，可配置是否归档。 |

## 11. 风险与缓解

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|----------|
| JSONL 格式随版本变化 | 中 | 高 | 解析层做版本适配，日志记录格式版本号 |
| Hook 触发不稳定 | 低 | 中 | 定时扫描作为兜底机制 |
| 会话数据量大导致解析慢 | 低 | 低 | 增量处理，仅解析新文件 |
| 状态推断不准确 | 高 | 低 | 标记为 `unknown`，不做强制判断 |
| 摘要质量不够好 | 中 | 中 | 后续可接入 LLM 做二次摘要 |

## 12. 总结

本方案的核心洞察：**不要重新发明轮子**。

Claude Code 已经提供了完整的会话数据存储，我们需要做的不是"捕获"数据，而是在此基础上构建 **解析 → 摘要 → 组织 → 检索** 的管道。

| 维度 | 原始需求 | 优化后方案 |
|------|----------|-----------|
| 数据获取 | 从插件底层实时捕获 | 直接读取已有 JSONL 文件 |
| 实时性 | 对话发生即记录 | 会话结束后异步处理 |
| 性能影响 | 可能影响编辑器 | 零影响（后处理） |
| 实现复杂度 | 高（需要逆向插件） | 低（文件读写 + 模板） |
| 维护成本 | 高（跟随插件版本） | 低（JSONL 格式相对稳定） |
